# CLAUDE.md - 项目约定

> 本文档定义项目的目录结构、工作流程和协作约定，确保任何接手者能快速理解项目。

---

## 1. 项目背景

仿猫眼电影票务平台，包含 C 端购票小程序、B 端管理后台、AI 智能助手三大模块。

| 模块 | 说明 |
|------|------|
| C 端小程序 | 用户购票、选座、支付、观影体验 |
| B 端管理后台 | 电影、影院、排片、订单、营销管理 |
| AI 智能助手 | 智能客服、智能推荐、Function Call 购票 |

详细需求见 `docs/requirements/` 目录。

---

## 2. 技术栈

| 层次 | 技术 | 说明 |
|------|------|------|
| 后端主服务 | Spring Boot | 核心业务逻辑 |
| AI 助手模块 | Spring Boot（独立服务） | AI 相关功能 |
| 前端 Web | Vue | Web 端 |
| 前端小程序 | UniApp | 小程序端 |

## 4. 进度管理

### 4.1 核心原则

**功能拆分，分 CC 开发，避免单窗口上下文爆炸。**

- 每个 CC 负责 1-3 个功能模块
- 新 session 读取进度文档接手开发

### 4.2 功能模块追踪

文件：`PROJECT_PLAN.md`

| 模块 | 状态 | 进度 | 关键文件 | 备注 |
|------|------|------|----------|------|
| 模块名 | 状态标识 | XX% | 文件路径 | 简要说明 |

### 4.3 任务状态标识

| 标识 | 含义 |
|------|------|
| ⏳ | 待开始 (Todo) |
| 🔵 | 进行中 (In Progress) |
| ✅ | 已完成 (Done) |
| ❌ | 阻塞 (Blocked) |

### 4.4 新 Session 接手流程

1. 读取 `CLAUDE.md` → 了解项目约定
2. 读取 `PROJECT_PLAN.md` → 了解当前进度
3. 找到 ⏳ 或 🔵 状态的模块，继续开发

## 5. 接口规范

### 5.1 RESTful API 约定

- **URL 格式**：小写，连字符分隔，名词复数
  - ✅ `/api/users`, `/api/order-items`
  - ❌ `/api/User`, `/api/orderItems`

- **HTTP 方法**：
  | 方法 | 用途 |
  |------|------|
  | GET | 查询 |
  | POST | 新增 |
  | PUT | 全量更新 |
  | PATCH | 部分更新 |
  | DELETE | 删除 |

### 5.2 统一返回格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

### 5.3 错误码约定

| 错误码 | 含义 |
|--------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 401 | 未授权 |
| 403 | 禁止访问 |
| 404 | 资源不存在 |
| 500 | 服务器错误 |
| 1xxx | 业务错误码（待细化） |

### 5.4 接口文档位置

`docs/design/api/` 目录下，按模块划分 API 文档。

## 6. 模块间通信规范

### 6.1 通信方式

**HTTP（RESTful API）**

| 通信链路 | 方式 | 说明 |
|----------|------|------|
| 前端 → 主服务 | HTTP | 业务接口调用 |
| AI 助手 → 主服务 | HTTP | 获取业务数据 |
| AI 助手 → 百炼平台 | API | 大模型能力 |

### 6.2 服务调用关系

```
┌─────────────────────────────────────────────────────┐
│                    前端层                            │
│         ┌─────────────┐    ┌─────────────┐         │
│         │  Vue Web    │    │  UniApp     │         │
│         └──────┬──────┘    └──────┬──────┘         │
└────────────────┼──────────────────┼────────────────┘
                 │ HTTP             │ HTTP
                 ▼                  ▼
┌─────────────────────────────────────────────────────┐
│                   后端层                             │
│         ┌─────────────────────────────────┐         │
│         │      Spring Boot 主服务          │         │
│         │      (业务逻辑)                  │         │
│         └──────────────▲──────────────────┘         │
│                        │ HTTP                       │
│                        │                            │
│         ┌──────────────┴──────────────────┐         │
│         │    Spring Boot AI 助手模块       │         │
│         │  (智能客服 + 智能推荐 + Fn Call)  │         │
│         └──────────────┬──────────────────┘         │
│                        │ API                         │
│                        ▼                            │
│         ┌─────────────────────────────────┐         │
│         │       百炼平台 API               │         │
│         │      (大模型能力)                │         │
│         └─────────────────────────────────┘         │
└─────────────────────────────────────────────────────┘
```

### 6.3 服务间调用规范

- **AI 助手 → 主服务**：通过 HTTP 调用主服务接口获取数据
- **AI 助手 → 百炼平台**：通过 API 调用大模型能力
- 接口地址配置在环境变量中
- 超时设置：建议 30s
- 错误处理：调用失败时应有重试或降级策略

## 9. Git 约定

### 9.1 提交格式

```
<type>: <description>
```

### 9.2 Type 类型

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `style` | 代码格式（不影响逻辑） |
| `refactor` | 重构 |
| `test` | 测试相关 |
| `chore` | 构建/工具相关 |

### 9.3 示例

```
feat: add user login module
```

## 11. 行为约束

### 11.1 新 Session 接手 SOP

**标准流程：**

```
Step 1: 读取 CLAUDE.md
        → 了解项目约定、技术栈、目录结构

Step 2: 读取 PROJECT_PLAN.md
        → 了解当前进度、各模块状态

Step 3: 找到待开发模块
        → 优先 ⏳ 待开始
        → 其次 🔵 进行中（接手未完成）

Step 4: 读取相关文档
        → docs/requirements/ 了解需求
        → docs/design/ 了解设计（如有）

Step 5: 开始开发
        → 开发前更新状态为 🔵
        → 完成后更新状态为 ✅
```

**快速恢复上下文的关键文件：**

| 优先级 | 文件 | 用途 |
|--------|------|------|
| P0 | `CLAUDE.md` | 项目约定 |
| P0 | `PROJECT_PLAN.md` | 当前进度 |
| P1 | `docs/requirements/` | 需求详情 |
| P1 | `docs/design/` | 设计详情 |

### 11.2 开发流程约束

#### 开发完整流程

```
开发开始 → 更新状态为 🔵 → 编写代码 → 自测 → 提交 Git → 更新状态为 ✅
```

#### 自测要求

**功能开发完成后，必须自测通过才能提交 Git。**

自测清单：

- [ ] 功能基本流程跑通
- [ ] 异常情况处理正确（参数错误、空值等）
- [ ] 接口返回格式符合规范
- [ ] 无明显 Bug

**自测记录：**

在 `docs/dev-logs/` 中简要记录自测结果：

```markdown
## YYYY-MM-DD 自测记录

**模块**：用户登录
**测试项**：
- ✅ 正常登录
- ✅ 密码错误提示
- ✅ 账号不存在提示
**结果**：通过
```

#### Git 提交约束

**代码改动后必须提交 Git。**

| 约束项 | 要求 |
|--------|------|
| 提交时机 | 自测通过后立即提交 |
| 提交格式 | `<type>: <description>`，详见第 9 章 |
| 提交粒度 | 按功能点提交，避免一次提交过多内容 |
| 提交前检查 | 确保不提交敏感配置（密码、Key 等） |

**禁止事项：**

- ❌ 代码改动后不提交 Git
- ❌ 提交信息模糊（如 "update", "fix bug"）
- ❌ 一次提交多个不相关功能

#### 改动必更新进度

**任何代码改动后，必须更新 PROJECT_PLAN.md。**

| 改动类型 | 更新内容 |
|----------|----------|
| 开始开发 | 状态改为 🔵 |
| 完成开发 | 状态改为 ✅，进度改为 100% |
| 遇到阻塞 | 状态改为 ❌，备注说明原因 |
| 部分完成 | 更新进度百分比，备注当前进展 |

**示例：**

```markdown
| 模块 | 状态 | 进度 | 关键文件 | 备注 |
|------|------|------|----------|------|
| 用户登录 | 🔵 | 60% | backend/main-service/auth | 接口完成，前端联调中 |
```

### 11.3 文档创建约束

**禁止随意创建文档，遵循以下原则：**

| 允许创建 | 需确认后创建 | 禁止创建 |
|----------|--------------|----------|
| `docs/dev-logs/` 下的开发日志 | 新的目录结构 | 与现有文档内容重复的文档 |
| `docs/qa/checklists/` 下的验收清单 | 修改 CLAUDE.md | 无明确用途的文档 |
| 必要的设计文档（API 设计等） | 修改 PROJECT_PLAN.md | 临时笔记类文档 |

**创建文档前自问：**

1. 是否已有类似文档？有则复用或更新
2. 是否属于已约定的目录结构？
3. 是否有明确用途？

**正确的做法：**

- 需要记录开发过程 → 写到 `docs/dev-logs/`
- 需要记录设计决策 → 写到 `docs/design/`
- 需要更新约定 → 先和用户确认，再改 `CLAUDE.md`
